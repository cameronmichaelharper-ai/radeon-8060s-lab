# Open risks

- transformers==5.3.0 is the ONLY 5.3.x release (5.3.1 is hallucinated and will fail 'No matching distribution'). If 5.3.0 ever conflicts with the installed torch/peft, relax to an unpinned current 5.x (>=5.3.0,<6.0.0) which colpali-engine 0.3.17 accepts -- e.g. `uv pip install "transformers>=5.3.0,<6.0.0"`.
- The load recipe deliberately DROPS device_map='cpu' and uses .to('cpu') so `accelerate` is NOT required. If you re-add device_map='cpu' anywhere, from_pretrained will raise ImportError until you also `uv pip install "accelerate>=1.1.0,<2.0.0"`.
- torch 2.8.0 on Windows PyPI is CPU-only (torch-2.8.0-cp312-cp312-win_amd64.whl); torch.cuda.is_available() will be False by design. Do NOT pass an --index-url pointing at a CUDA channel -- there is no NVIDIA GPU/CUDA on this AMD gfx1151 machine and a CUDA wheel would fail to import. No ROCm/WSL path is used or needed here (known dead end).
- CPU-only inference on colSmol-256M is slow but fine for a few dozen pages. BATCH_SIZE defaults to 1 for RAM safety; if you fall back to colSmol-500M or colpali-v1.3 (PaliGemma-3B) on this 128GB box, encoding is much slower and RAM heavier -- keep BATCH_SIZE=1 and expect minutes per page for the 3B model. colpali-v1.3 also requires switching to ColPali/ColPaliProcessor classes (not ColIdefics3) in 02/03.
- Docker Desktop bind mount: -v C:/AI/research/vision-retrieval-poc/qdrant_storage:/qdrant/storage needs that path enabled in Docker Desktop file sharing. If the mount silently fails (empty/read-only storage), either enable the path, use a native Docker named volume (`-v qdrant_storage:/qdrant/storage`), or drop -v entirely for this throwaway PoC (data is non-precious and re-indexable).
- Europe PMC ?pdf=render is the download path (verified 200 for all 3 IDs on 2026-07-03). If it returns non-PDF bytes (endpoint outage), 01 raises a clear error. Manual fallback: fetch the oa_package .tar.gz via ftplib against ftp.ncbi.nlm.nih.gov using the ftp:// href read from oa.fcgi (the tgz path is per-record and must NOT be constructed by hand), then extract the .pdf member.
- First run of 02/03 downloads model weights from HuggingFace (colSmol-256M, ungated). Needs network + a few GB HF cache. If HF is unreachable, from_pretrained fails; pre-seed the cache or set HF_HOME to a warm cache.
- PATCH_DIM is derived at runtime from the first embedding (colSmol projects to 128), so a model swap won't desync the Qdrant collection dim -- but 02 always recreates the collection, so re-run 02 after any model change to avoid a dim mismatch on upsert.
- hnsw_config m=0 makes the multivector field brute-force/rerank-only -- correct and RAM-cheap for this tiny corpus. If you later scale up and want ANN prefiltering you'd add a separate pooled single-vector field; not needed here.
- System Python is 3.14.5 (too new for these ML wheels); the venv MUST be pinned to 3.12 via `uv venv --python 3.12`. If uv can't find a 3.12 interpreter, run `uv python install 3.12` first.
